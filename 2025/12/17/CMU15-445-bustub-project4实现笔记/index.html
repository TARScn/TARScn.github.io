锁管理器

为了确保事务操作的正确交错，数据库管理系统（DBMS）使用锁管理器（LM）来控制事务何时可以访问数据项。锁管理器（LM）的基本思想是它维护一个关于活动事务当前持有的锁的内部数据结构。事务在访问数据项之前向 LM 发出锁请求，LM 将要么授予锁，要么阻塞事务直到锁可用，要么中止事务。

这个任务的核心是实现一个全局锁管理器 (Lock Manager, LM)，用于在事务访问数据时控制并发，通过两阶段锁协议 (2PL) 保证事务的隔离性。

关键数据结构与状态

需要维护的数据主要位于 Transaction 类和 LockManager 类中。

A. 锁请求队列 (LockRequestQueue)

  • 每个TableOID或RID都会映射到一个LockRequestQueue。

  • 组成:

    • std::list: 等待或持有该资源锁的请求列表。

    • std::condition_variable: 用于唤醒被阻塞的事务。

    • 核心逻辑: 当一个锁释放时，需要通过 CV 通知等待队列中的事务检查是否可以获取锁。

B. 事务状态 (Transaction State)

在 Transaction 类中维护，LM 必须根据操作更新它：

  • GROWING (增长阶段): 事务正在获取锁。

  • SHRINKING (收缩阶段): 事务释放了第一个锁之后进入此阶段（严格 2PL 下通常在 Commit/Abort 时才释放，但不同隔离级别有区别）。

  • ABORTED: 发生非法锁操作时。

    • 注意: 如果非法操作，除了设为 ABORTED，还需抛出异常。

    • 如果只是获取锁失败（如死锁），返回 false，不抛异常。

C. 隔离级别行为 (Isolation Levels)

LM 的行为（能否加锁、何时解锁）取决于隔离级别：

隔离级别 读锁 (S/IS) 写锁 (X/IX) 2PL 行为
REPEATABLE_READ 需要获取，直到事务结束 需要获取，直到事务结束 严格遵循 2PL (Growing -> Shrinking)
READ_COMMITTED 需要获取，用完即放 需要获取，直到事务结束 读锁可能会提前释放 (打破严格 2PL)
READ_UNCOMMITTED 不需要获取 需要获取，直到事务结束 只加写锁，不加读锁

需要实现的接口 (API)

A. LockTable(Transaction, LockMode, TableOID)

  • 作用: 申请表级锁。

  • 逻辑:

  1. 检查合法性:

    • 如果是 READ_UNCOMMITTED 且申请 S/IS/SIX 锁 -> Abort (RU 模式不加读锁)。

    • 如果是 SHRINKING 阶段且申请锁 -> Abort (违反 2PL)。

    • (特例: READ_COMMITTED 在 Shrinking 阶段能不能加锁？通常不能，除了某些特定的锁升级场景，需仔细看测试用例)。

  2. 获取请求队列: 找到对应 TableOID 的队列。

  3. 锁升级 (Upgrade): 检查该事务是否已在该表持有锁。如果持有，判断是否需要升级 (例如 IS -> S, S -> X)。升级通常有优先级。

  4. 阻塞等待: 如果当前锁与队列中已有的锁不兼容，调用 cv.wait() 阻塞。

  5. 授予锁: 更新 request->granted_ = true，更新事务的 lock_set。

B. UnlockTable(Transaction, TableOID)

  • 作用: 释放表级锁。

  • 逻辑:

  1. 检查行锁: 只有当该表下的所有行锁都释放后，才能释放表锁（意向锁协议）。

  2. 更新阶段: 如果释放的是 X/IX 锁（或者在 RR 模式下释放 S 锁），将事务状态从 GROWING 改为 SHRINKING。

  3. 清理: 从队列中移除请求，从事务 lock_set 中移除。

  4. 唤醒: 调用 cv.notify_all() 唤醒等待该表的其他事务。

C. LockRow(Transaction, LockMode, TableOID, RID)

  • 作用: 申请行级锁。

  • 逻辑:

  1. 检查表锁: 事务必须先持有该表的意向锁 (IS/IX/SIX) 才能申请行锁。如果没有 -> Abort。

  2. 其余逻辑: 类似 LockTable (检查 2PL 阶段，处理等待，处理升级)。

D. UnlockRow(Transaction, TableOID, RID, force)

  • 作用: 释放行级锁。

  • 参数 force:

  1. 如果 force == true: 忽略 2PL 规则检查，直接解锁，不改变事务状态（不进入 Shrinking）。这通常用于执行器只是“看一眼”元组但并不真的处理它的情况。

  2. 如果 force == false: 正常逻辑，可能会触发生命周期状态变更 (Growing -> Shrinking)。

代码实现

  1. 核心数据结构设计
    锁管理器通过哈希表和队列来管理不同资源上的锁竞争。
  • 分层结构：
    • table_lock_map_：维护表级锁（OID -> 锁请求队列）。
    • row_lock_map_：维护行级锁（RID -> 锁请求队列）。
  • 请求队列 (LockRequestQueue)：
    • 每个资源（表或行）都有一个队列。
    • request_queue_：使用 std::list 存储等待或已持有的 LockRequest，通常遵循 FIFO (先进先出) 原则，但受兼容性限制。
    • cv_ (Condition Variable)：核心同步机制。当锁被释放时，利用条件变量唤醒等待该资源的所有事务。
    • upgrading_：专门用于记录当前正在尝试锁升级的事务 ID，防止多个事务同时升级导致死锁。

  1. 加锁逻辑 (LockTable / LockRow)
    加锁的核心思想是 “检查-等待-授权” (Check-Wait-Grant) 模型。

    1. 预检查 (Pre-checks)：
      • 离级别限制：
        • READ_UNCOMMITTED 试图获取共享锁 (S/IS/SIX) 会直接抛出异常并 Abort。
        • 收缩阶段检查：如果事务处于 SHRINKING 阶段（即已经释放过锁），除了 READ_COMMITTED 下的特殊情况外，通常禁止再获取新锁（严格遵循 2PL）。
      • 层级约束 (LockRow 特有)：在 LockRow 中，必须调用 CheckAppropriateLockOnTable 确保事务已经持有了正确的表级意向锁，才能申请行锁。
    2. 锁升级处理 (CheckLockUpdate…)：
      • 如果事务已持有该资源的锁，但请求更高级别的锁（例如从 S 升级到 X），代码会优先处理。
      • 逻辑：释放旧锁 -> 修改请求为新模式 -> 插入到队列适当位置（通常保持优先级）-> 尝试授权。
      • 防死锁：利用 queue->upgrading_ 标记，同一资源同一时间只允许一个事务升级。
    3. 等待与授权循环：
      • 如果没有直接获得锁，请求被加入队列。
      • 使用 cv_.wait(lk, predicate) 进入阻塞等待。
      • Predicate (唤醒条件)：传入的 lambda 表达式会调用 GrantNewLocksIfPossible。只有当 request->granted_ 变为 true 时，事务才会被唤醒并继续执行。
      • GrantNewLocksIfPossible：这是调度的核心。它遍历队列，检查队首请求，并根据 兼容性矩阵 (AreLocksCompatible) 判断后续请求是否可以与当前持有者共存（例如多个 S 锁可以共存，但 X 锁必须独占）。

  2. 解锁逻辑 (UnlockTable / UnlockRow)
    解锁的核心思想是 “状态流转与唤醒”。

    1. 状态流转 (2PL Phase Change)：
      • 当释放锁时，代码根据隔离级别判断是否需要将事务状态从 GROWING 切换到 SHRINKING。
      • 例如：在 REPEATABLE_READ 下，一旦释放 S 或 X 锁，事务即进入收缩阶段，之后不能再加锁。
    2. 强制解锁 (force 参数)：
      • UnlockRow 包含一个 force 参数。这通常用于执行器（Executor）层，当只需要“看一眼”数据（如做简单的过滤）而不需要持有锁直到事务结束时，可以强制解锁而不改变事务状态（不进入 Shrinking）。
    3. 清理与通知：
      • 从队列中移除请求。
      • 从事务的 lock_set 中移除记录。
      • 关键步骤：调用 list->cv_.notify_all()。这会唤醒所有在 LockTable/Row 中沉睡的线程，让它们重新竞争锁（通过 GrantNewLocksIfPossible 检查）。

  3. 死锁检测 (Deadlock Detection)
    代码中包含了一个后台线程 RunCycleDetection，其实现思想是 基于图的环检测。

    1. 构建等待图 (Waits-for Graph)：
      • 遍历所有锁队列。
      • 如果事务 A 持有锁 (granted_ = true)，事务 B 在等待锁 (granted_ = false)，则添加一条边 B -> A (B 等待 A)。
    2. 环检测 (DFS)：
      • 使用深度优先搜索 (Dfs) 查找图中是否存在环。
      • 如果存在环（例如 A 等 B，B 等 A），说明死锁。
    3. 解除死锁：
      • 策略是 “杀掉最年轻的事务”。
      • HasCycle 会返回环中 txn_id 最大的事务（即最晚开始的事务）。
      • 调用 txn_manager_->Abort() 中止该事务，从而释放它持有的锁，打破环。
